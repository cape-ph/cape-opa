## 1. Artifact Storage Location

**Question**: Where should cape-opa release artifacts be stored?

**Current Assumption**: GitHub Releases (attached to git tags)

**Options**:

### Option A: GitHub Releases (Assumed)
**Pros**:
- Simple integration with GitHub Actions
- Version control tied to git tags
- Public access (if repo is public)
- Free for public repos

**Cons**:
- Rate limits (60 requests/hour unauthenticated)
- Requires GitHub token for private repos
- External dependency for Pulumi downloads

**URLs**:
```
https://github.com/cape-ph/cape-opa/releases/download/v1.2.3/policy-bundle-v1.2.3.tar.gz
https://github.com/cape-ph/cape-opa/releases/download/v1.2.3/data-bundle-generator-v1.2.3.zip
```

### Option B: S3 Bucket
**Pros**:
- No rate limits
- Faster in AWS region
- S3 versioning available
- Direct control

**Cons**:
- Additional S3 bucket to manage
- IAM permissions needed
- Need to sync from GitHub Actions

**URLs**:
```
https://cape-artifacts.s3.amazonaws.com/opa/v1.2.3/policy-bundle.tar.gz
https://cape-artifacts.s3.amazonaws.com/opa/v1.2.3/data-bundle-generator.zip
```

### Option C: Artifact Repository (Enterprise)
**Pros**:
- Enterprise features (Artifactory, Nexus)
- Access control
- Security scanning
- Audit logs

**Cons**:
- Additional infrastructure
- Cost
- Complexity

**Decision Needed**: Which option?

**Impact**:
- Pulumi `RemoteArchive` URL
- Ansible `get_url` URLs
- GitHub Actions upload step
- IAM permissions

**Recommendation**: Start with Option A (GitHub Releases) for simplicity, migrate to Option B if rate limits become an issue.

## 2. Testing Strategy

**Question**: How comprehensive should testing be?

**Options Presented**: See 08-testing.md for full details

### Option 1: Minimal (Current)
- OPA syntax tests only
- No Lambda tests
- Manual verification

### Option 2: Unit Tests
- OPA policy tests with fixtures
- Lambda unit tests with mocks
- CI/CD integration

### Option 3: Full Integration
- Docker Compose test environment
- Real PostgreSQL, OPA, LocalStack S3
- End-to-end testing

**Trade-offs**:
- Time to implement vs. confidence in changes
- CI/CD speed vs. coverage
- Maintenance burden vs. catching bugs early

**Decision Needed**: Which option to implement first?

**Recommendation**: Start with Option 1, evolve to Option 2 within 1-2 sprints, consider Option 3 when stability is critical.

## 3. Seed Data Management

**Question**: Should any authorization data be version-controlled?

**Current State**: All data is dynamic (generated by Lambda from database)

**Future Need**: May need versioned data for:
- Auditing requirements
- Regulatory compliance
- Reproducible testing
- Historical analysis
- Demo environments

**Potential Approach**:
```
cape-opa/
├── policies/  (version controlled)
└── seed/      (NEW - version controlled data)
    ├── demo/
    │   ├── users.json
    │   ├── tributaries.json
    │   └── resources.json
    └── test/
        └── (test fixtures)
```

**Bundle Generator Would**:
1. Load seed data from repository
2. Query database for dynamic data
3. Merge seed + dynamic data
4. Generate bundle

**Pros**:
- Reproducible environments
- Version-controlled baseline data
- Easier testing

**Cons**:
- Data duplication (seed vs. database)
- Sync complexity
- May not match current database state

**Decision Needed**: 
- Is seed data needed?
- What data should be seeded?
- How to handle seed + dynamic merge?

**Recommendation**: Defer until customer/compliance requirements become clear.

## 4. OPA Configuration Location

**Question**: Where should OPA server configuration live?

**Current State**: OPA config in cape-cod Pulumi (user-data script)

**Alternative**: Move to cape-opa repository

**Pros of Moving**:
- All OPA concerns in one repo
- Config versioned with policies
- Config changes don't require Pulumi
- Easier to test config changes

**Cons of Moving**:
- Ansible must deploy config
- More complex initial setup
- Config separated from infrastructure

**Potential Structure**:
```
cape-opa/
├── policies/
├── bundle-generator/
└── config/
    ├── opa-config.yaml.j2  (Jinja2 template)
    └── README.md
```

**Migration Path**:
1. Create config template in cape-opa
2. Include in release artifacts
3. Ansible downloads and renders template
4. Ansible deploys to OPA EC2
5. Remove from Pulumi user-data

**Decision Needed**: Should we move config? When?

**Recommendation**: Defer until OPA config needs frequent changes or becomes complex.

## 5. Multi-Environment Strategy

**Question**: How to handle dev/staging/prod differences?

**Current Approach**: Single set of variables, overrides per environment

**Considerations**:

### Lambda Schedule
- Dev: `rate(5 minutes)` - Fast feedback
- Staging: `rate(10 minutes)`
- Prod: `rate(15 minutes)` - Reduce load

### Lambda Resources
- Dev: 512 MB memory, 5 min timeout
- Prod: 1024 MB memory, 10 min timeout

### Policy Versions
- Dev: Latest (main branch)
- Staging: Release candidate (v1.2.3-rc1)
- Prod: Stable release (v1.2.3)

**Decision Needed**:
- Should environments run different OPA versions?
- How to handle promotion (dev → staging → prod)?
- Automatic or manual promotion?

**Recommendation**: Start with same version everywhere, add promotion workflow when team grows.

## 6. Monitoring and Alerting

**Question**: What should we monitor and alert on?

**Potential Metrics**:

### Lambda Metrics
- Invocation count (should be consistent)
- Error rate (should be near zero)
- Duration (should be stable)
- Bundle size (should grow slowly)

### OPA Metrics
- Authorization decision latency
- Deny rate (sudden changes may indicate policy issues)
- Bundle reload failures

### S3 Metrics
- Bundle upload frequency
- Bundle file sizes
- Storage costs

**Potential Alerts**:
- Lambda failure rate > 5%
- Lambda duration > 4 minutes
- No bundle uploaded in 1 hour
- OPA can't load bundle

**Decision Needed**:
- Which metrics to collect?
- Which alerts to configure?
- Who gets alerted?

**Recommendation**: Start with basic CloudWatch alarms, expand based on operational experience.

## 7. Backward Compatibility

**Question**: How long to support old policy/data formats?

**Scenarios**:
- Change data bundle structure
- Change policy decision format
- Change Lambda environment variables

**Potential Policy**:
- **Option A**: Breaking changes allowed, coordinated deployment
- **Option B**: Maintain compatibility for N versions (e.g., 3 releases)
- **Option C**: Never break compatibility, always additive

**Decision Needed**: What's our backward compatibility policy?

**Recommendation**: Option B - Maintain compatibility for 2-3 releases, allowing gradual migration.

## 8. Security Scanning

**Question**: Should we scan artifacts for vulnerabilities?

**Options**:

### Policy Scanning
- OPA has built-in linting
- Could add custom security rules

### Lambda Dependency Scanning
- Snyk, Trivy, or GitHub Dependabot
- Scan Python dependencies for CVEs
- Block releases with critical vulnerabilities

### Infrastructure Scanning
- Scan Pulumi code for misconfigurations
- Check IAM permissions are least-privilege

**Decision Needed**:
- Which scanners to use?
- Block releases on findings or warn only?
- Who is responsible for remediation?

**Recommendation**: Enable GitHub Dependabot for Python dependencies, add scanning later as needed.
